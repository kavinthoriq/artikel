Seiring berkembangnya ukuran tim dan kompleksitas aplikasi web, arsitektur monolitik frontend seringkali menjadi botol leher (bottleneck). Di sinilah Micro-Frontend hadir sebagai penyelamat.

Apa itu Micro-Frontend?

Istilah Micro-Frontend pertama kali muncul pada akhir 2016. Konsep ini pada dasarnya adalah perpanjangan dari prinsip microservices yang diterapkan pada dunia frontend. Alih-alih memiliki satu codebase besar yang dikelola oleh semua tim, aplikasi dibagi menjadi fitur-fitur independen.

"Micro-frontends are an architectural style where independently deliverable frontend applications are composed into a greater whole." - Cam Jackson, MartinFowler.com

Kenapa Harus Migrasi?

Jika Anda bekerja dalam tim kecil dengan proyek yang relatif sederhana, monolitik mungkin adalah pilihan terbaik. Namun, ketika jumlah engineer melebihi 20 orang dan mereka mengerjakan bagian aplikasi yang sama, masalah mulai muncul:

  • Deploy conflicts: Satu bug kecil dapat menggagalkan deploy seluruh aplikasi.
  • Tech stack lock-in: Sulit untuk mencoba teknologi baru (misal: migrasi dari React 16 ke 18) karena harus mengubah seluruh aplikasi.
  • Slow builds: Waktu build dan CI/CD menjadi sangat lambat (bisa mencapai >30 menit).

Arsitektur Teknis

Ada beberapa pendekatan untuk mengimplementasikan micro-frontend, mulai dari iframe, NGINX routing, hingga pendekatan modern berbasis JavaScript.

Webpack Module Federation

Sejak rilis Webpack 5, Module Federation menjadi standar de-facto untuk membangun micro-frontend. Fitur ini memungkinkan beberapa build JavaScript (aplikasi terpisah) untuk saling membagikan modul secara dinamis pada saat runtime.

Contoh Implementasi Kode

Berikut adalah konfigurasi dasar webpack.config.js untuk aplikasi Host (yang membungkus aplikasi lain): 

const ModuleFederationPlugin = require("webpack/lib/container/ModuleFederationPlugin");

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: "host_app",
      remotes: {
        cart: "cart@http://localhost:3002/remoteEntry.js",
        products: "products@http://localhost:3001/remoteEntry.js",
      },
      shared: ["react", "react-dom"],
    }),
  ],
};

Dan berikut adalah konfigurasi untuk aplikasi Remote (misalnya aplikasi Keranjang/Cart): 

const ModuleFederationPlugin = require("webpack/lib/container/ModuleFederationPlugin");

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: "cart",
      filename: "remoteEntry.js",
      exposes: {
        "./CartWidget": "./src/components/CartWidget",
      },
      shared: ["react", "react-dom"],
    }),
  ],
};

Dengan konfigurasi di atas, aplikasi Host dapat me-render komponen CartWidget seolah-olah itu adalah komponen lokal, padahal dikembangkan dan di-deploy secara terpisah oleh tim yang berbeda.

Kesimpulan

Micro-Frontend bukanlah silver bullet (solusi ajaib untuk semua masalah). Ia datang dengan kompleksitas infrastruktur tambahan. Pastikan masalah yang Anda hadapi adalah masalah organisasi dan skalabilitas tim, bukan sekadar mengikuti tren teknologi terbaru.
